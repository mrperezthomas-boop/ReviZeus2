RÉVIZEUS — BLOC E — QUIZ NOUVELLE GÉNÉRATION + CORRECTIONS + FEEDBACK RÉSULTAT
VERSION : EXECUTION BLOCK
DATE : 2026-03-24
SOURCE : REVIZEUS_DECOUPAGE_PAR_BLOCS_POUR_CODAGE_IA.txt + document maître fusionné
OBJECTIF : fiabiliser totalement la génération des quiz, la qualité des corrections divines et la mise en scène des résultats afin que toute la boucle pédagogique et RPG repose sur quelque chose de crédible, varié, lisible, motivant et extensible.

====================================================================
0 — POSITION DU BLOC E DANS LE CHANTIER GLOBAL
====================================================================

Ce bloc arrive après :
- BLOC A — stabilisation technique et socle transverse ;
- BLOC B — dialogues RPG universels et feedbacks immersifs ;
- BLOC C — audio global, TTS, ducking, loadings divins, pattern vidéo → image fixe ;
- BLOC D — Oracle premium + pipeline résumé → validation → temple → quiz.

Raison :
- le quiz est le point où l’IA, l’audio, la pédagogie, les récompenses et les dialogues divins se rencontrent ;
- si les couches A à D ne sont pas au moins amorcées, les quiz risquent d’être techniquement instables, pédagogiquement faibles ou incohérents avec les savoirs validés ;
- le bloc E prépare directement les blocs F, G, H, L, M et O ;
- si le quiz n’est pas robuste, tout le reste du jeu perd sa crédibilité : progression, récompenses, badges, analytics, aventure, boss et temples deviennent artificiels.

Le bloc E doit donc être traité comme un chantier cœur, pas comme une simple amélioration de prompts.

====================================================================
1 — OBJECTIF FINAL
====================================================================

À la fin du bloc E, RéviZeus doit pouvoir :
- générer des quiz réellement crédibles, variés et adaptés ;
- éviter les patterns toxiques type “réponse B trop souvent correcte” ;
- produire des distracteurs plausibles, ni absurdes ni trop faciles ;
- adapter le ton et la difficulté selon l’âge, la classe, la matière, l’humeur, le profil d’apprentissage et le contexte d’entrée ;
- proposer de vraies corrections maïeutiques, utiles et non humiliantes ;
- générer un mnémonique pertinent après erreur quand c’est utile ;
- expliquer pourquoi la bonne réponse est bonne et pourquoi les autres ne le sont pas ;
- lire la question, les réponses et les corrections via TTS avec ducking audio ;
- mettre en scène un résultat final premium et mémorable ;
- fournir une base de vérité pédagogique sur laquelle pourront s’appuyer les récompenses, les temples, le moteur cognitif, les badges, les quêtes et l’aventure.

Le ressenti visé est :
- sérieux dans le contenu ;
- noble dans la mise en scène ;
- motivant dans le feedback ;
- juste dans la difficulté ;
- profondément cohérent avec le dieu, le savoir et le profil de l’utilisateur.

====================================================================
4 — VISION PRODUIT DU FLOW QUIZ
====================================================================

Le flow cible devient :

Entrée dans le quiz
→ introduction divine éventuelle
→ question affichée clairement
→ TTS optionnel question/réponses
→ choix de la réponse
→ feedback de validation court
→ correction maïeutique si nécessaire
→ mnémonique ou approfondissement si utile
→ enchaînement vers la question suivante
→ fin de session
→ résultat premium séquencé
→ message divin adapté
→ application propre des rewards via le bloc F ensuite

Règle fondamentale :
le quiz ne doit jamais être ressenti comme un tirage aléatoire de QCM plats.
Il doit être perçu comme une épreuve divine juste, lisible, variée et intelligemment construite.

5 — SOUS-CHANTIERS DÉTAILLÉS
====================================================================

--------------------------------------------------------------------
E1 — GÉNÉRATION QUIZ ROBUSTE ET JSON STRICT
--------------------------------------------------------------------

Objectif
Empêcher les quiz incohérents, mal formés, incomplets ou pauvres pédagogiquement.

À faire
- renforcer les prompts Gemini pour imposer une structure JSON stricte et validable ;
- imposer un nombre clair de questions selon le contexte ;
- imposer un format stable pour chaque question : énoncé, options, index correct, explication, mnémonique éventuel, type de question, difficulté estimée ;
- mettre en place une validation post-génération locale : si JSON invalide, si options insuffisantes, si index impossible, si doublons trop forts, régénération contrôlée ;
- s’assurer que le quiz généré depuis un savoir parte du résumé validé et non d’un état intermédiaire ;
- distinguer quiz Oracle, quiz entraînement matière et quiz entraînement ultime si la logique métier le demande déjà.

Résultat attendu
Le moteur génère des quiz exploitables sans bricolage manuel ou patchs permanents dans l’UI.

--------------------------------------------------------------------
E2 — VARIÉTÉ DE TYPES DE QUESTIONS
--------------------------------------------------------------------

Objectif
Éviter la monotonie et adapter le style de question à la matière et au type de savoir.

Types à intégrer ou préparer
- définition ;
- compréhension ;
- calcul ;
- chronologie ;
- analogie ;
- vrai/faux ;
- cas pratique ;
- reconnaissance d’erreur ;
- logique étape par étape ;
- relation cause/effet ;
- classification ;
- mise en contexte.

À faire
- enrichir les prompts pour demander explicitement plusieurs types ;
- moduler les types par matière :
  - maths → calcul, logique, étape par étape ;
  - français → compréhension, reformulation, sens ;
  - histoire → chronologie, contexte, causalité ;
  - SVT → systèmes, cycles, fonctions ;
  - langues → sens, traduction, usage ;
- éviter d’avoir 5 questions quasi identiques dans une même session ;
- stocker si possible le type de question pour analytics futurs.

Résultat attendu
Une session donne le sentiment d’une vraie épreuve intelligente, pas d’une répétition de variantes plates.

--------------------------------------------------------------------
E3 — RANDOMISATION RÉELLE DES BONNES RÉPONSES
--------------------------------------------------------------------

Objectif
Supprimer le biais de position qui détruit la confiance dans le quiz.

À faire
- randomiser réellement l’ordre des options après génération ;
- recalculer l’index correct proprement ;
- vérifier la distribution sur une session entière ;
- empêcher des séquences trop suspectes du type même position correcte quatre fois de suite si cela peut être évité sans tricher avec le contenu ;
- conserver une logique transparente dans le code pour debug si besoin.

Résultat attendu
L’utilisateur ne peut plus détecter de pattern structurel grossier dans la position de la bonne réponse.

--------------------------------------------------------------------
E4 — DISTRACTEURS CRÉDIBLES ET NON RIDICULES
--------------------------------------------------------------------

Objectif
Faire en sorte que les mauvaises réponses soient pédagogiquement pertinentes.

À faire
- demander à Gemini des distracteurs plausibles et proches d’erreurs humaines réelles ;
- interdire les options absurdes, hors sujet ou trop évidemment fausses ;
- pour certaines matières, forcer des distracteurs issus de confusions classiques ;
- vérifier localement qu’il n’y a pas d’option dupliquée ou quasi identique à la bonne réponse ;
- conserver si possible un champ “reasoning trap type” ou équivalent dans les métadonnées debug pour usage ultérieur.

Résultat attendu
La mauvaise réponse révèle quelque chose d’utile sur la compréhension du joueur.

--------------------------------------------------------------------
E5 — ADAPTATION AU CONTEXTE JOUEUR
--------------------------------------------------------------------

Objectif
Faire varier difficulté, ton et densité selon le joueur réel.

Entrées à exploiter si disponibles
- âge ;
- classe / niveau scolaire ;
- humeur ;
- profil d’apprentissage ;
- matière ;
- type de quiz ;
- performance précédente ;
- éventuel score cognitif si déjà amorcé.

À faire
- enrichir les prompts pour que les questions soient adaptées à la tranche d’âge ;
- éviter le jargon excessif pour les plus jeunes ;
- rendre le ton plus progressif si profil fragile ;
- rendre le défi plus noble si profil challenge ;
- préparer l’interface avec le futur adaptive difficulty sans surcharger le bloc E.

Résultat attendu
Le quiz paraît adressé à l’utilisateur, pas généré pour un joueur abstrait.

--------------------------------------------------------------------
E6 — CORRECTION DIVINE MAÏEUTIQUE
--------------------------------------------------------------------

Objectif
Transformer la correction en vraie aide pédagogique.

À faire
- au lieu de simplement afficher la bonne réponse, générer une correction guidante ;
- faire parler le dieu de la matière avec un ton cohérent ;
- expliquer le raisonnement ou l’indice clé ;
- éviter tout ton humiliant ;
- proposer une forme courte par défaut et une forme enrichie si l’utilisateur veut approfondir ;
- intégrer une logique “pourquoi c’est juste” et “pourquoi les autres ne le sont pas”.

Résultat attendu
Même une erreur devient une progression utile et motivante.

--------------------------------------------------------------------
E7 — MNÉMONIQUES ET APPROFONDISSEMENTS
--------------------------------------------------------------------

Objectif
Rendre l’erreur mémorable et exploitable dans le temps.

À faire
- générer un mnémonique bref après certaines erreurs ;
- l’adapter au dieu, à l’âge et à la matière ;
- éviter les mnémoniques inutiles, trop artificiels ou trop longs ;
- proposer un bouton ou une option “approfondir” si le joueur veut comprendre davantage ;
- préparer le lien futur avec GeneratedArtifact, export et transformations divines du bloc I/J.

Résultat attendu
La correction ne disparaît pas instantanément : elle laisse une trace mentale exploitable.

--------------------------------------------------------------------
E8 — TTS QUIZ ET CORRECTIONS
--------------------------------------------------------------------

Objectif
Rendre l’expérience accessible, confortable et premium.

À faire
- ajouter ou fiabiliser le speaker sur la question ;
- permettre la lecture des réponses ;
- permettre la lecture de la correction ;
- assurer le ducking propre de la BGM ;
- permettre arrêt / reprise / nouvelle lecture sans bug ;
- éviter les overlaps de TTS entre question et correction ;
- garantir que l’icône speaker soit visible en entier et bien cadrée dans le layout ;
- gérer correctement le lifecycle.

Résultat attendu
Le TTS devient une couche naturelle du quiz, pas un gadget fragile.

--------------------------------------------------------------------
E9 — TRANSITIONS ET MICRO-FEEDBACKS PENDANT LE QUIZ
--------------------------------------------------------------------

Objectif
Rendre le quiz vivant sans le surcharger.

À faire
- feedback visuel propre au clic réponse ;
- retour sonore cohérent succès / erreur / validation ;
- animation discrète de passage à la question suivante ;
- petite respiration entre correction et question suivante ;
- cohérence audio avec le bloc C ;
- cohérence dialogue avec le bloc B.

Résultat attendu
La session garde un rythme noble, lisible et satisfaisant.

--------------------------------------------------------------------
E10 — RÉSULTAT PREMIUM ET RÉVÉLATION SÉQUENCÉE
--------------------------------------------------------------------

Objectif
Faire du résultat un vrai moment de récompense et de lecture de performance.

À faire
- animation des étoiles ;
- révélation séquentielle XP puis monnaies puis fragments puis badges si disponibles ;
- message divin final adapté à la performance ;
- BGM de résultat selon la qualité de la session ;
- si level-up, mise en scène dédiée ;
- si badge débloqué, révélation propre ;
- si analytics disponibles, afficher un ou deux verdicts réellement utiles ;
- préparer la comparaison future avec session précédente sans forcer une dépendance lourde.

Résultat attendu
La fin de quiz devient mémorable, compréhensible et motivante.

--------------------------------------------------------------------
E11 — COHÉRENCE ENTRE QUIZ ORACLE, ENTRAÎNEMENT MATIÈRE ET ENTRAÎNEMENT ULTIME
--------------------------------------------------------------------

Objectif
Éviter trois systèmes divergents qui finissent par se contredire.

À faire
- identifier les différences légitimes entre les modes ;
- factoriser la logique commune quand possible ;
- conserver les spécificités sans dupliquer toute la mécanique ;
- vérifier que les rewards et les fragments respectent la règle produit déjà posée ;
- préparer l’extension vers boss, défis et aventure.

Résultat attendu
Les différents quiz appartiennent au même écosystème, avec des nuances lisibles mais une base commune saine.

--------------------------------------------------------------------
E12 — ANALYTICS DE SESSION ET TRACES UTILES
--------------------------------------------------------------------

Objectif
Poser une base fiable pour le moteur cognitif, les recommandations et les verdicts.

À faire
- journaliser proprement le type de question, le temps de réponse, le bon/faux, la matière, le contexte du quiz, l’usage du TTS, l’usage d’aides éventuelles ;
- préparer des points d’entrée propres vers UserAnalyticsEngine ;
- distinguer ce qui est métier stable et ce qui relève du debug ;
- éviter de disperser des calculs analytics dans les Activities.

Résultat attendu
Le bloc E nourrit proprement les futurs blocs L et M.
