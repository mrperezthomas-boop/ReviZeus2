0 — POSITION DU BLOC D DANS LE CHANTIER GLOBAL
====================================================================

Ce bloc arrive après :
- BLOC A — stabilisation technique et socle transverse ;

====================================================================
1 — OBJECTIF FINAL
====================================================================

Faire de l’Oracle le cœur premium réel de RéviZeus.

À la fin du bloc D, le flow Oracle doit permettre à l’utilisateur de :
- entrer par trois vraies portes principales, plus une couche de conseil divine ;
- scanner un document, importer un document, ou créer un savoir depuis une demande ;
- recevoir un résumé lisible, structuré, adapté à l’âge et au contexte ;
- entendre ou relire ce résumé ;
- éditer ou reformuler ce résumé avant sauvegarde ;
- confirmer manuellement le temple / la matière de destination même si l’IA a détecté une matière ;
- sauvegarder un savoir propre avec métadonnées riches ;
- générer un quiz à partir du résumé validé, jamais à partir d’un état intermédiaire sale ;
- ne jamais voir un résultat brouillon, partiel ou incohérent.

Le ressenti premium visé est :
- spectaculaire à l’entrée ;
- fluide pendant l’usage ;
- noble pendant les chargements ;
- clair pendant la validation ;
- cohérent avec les dieux, l’audio, la DA et la logique pédagogique.

====================================================================
4 — VISION PRODUIT DU FLOW ORACLE
====================================================================

Le flow cible devient :

Entrée Oracle
→ choix d’une porte
→ capture / import / texte libre / conseil
→ extraction ou structuration matière première
→ pré-analyse matière + qualité
→ loading divin
→ génération du résumé structuré
→ proposition de titres
→ écran de validation du résumé
→ édition / reformulation / changement temple / confirmation
→ sauvegarde atomique du savoir
→ proposition ou lancement du quiz à partir du résumé validé
→ transition propre vers le résultat ou le temple

Règle fondamentale :
aucune donnée sale ne doit devenir l’état canonique du savoir.

Le savoir canonique doit être :
- le résumé validé ;
- le temple validé ;
- le titre validé ;
- les métadonnées d’entrée correctement stockées.

5 — SOUS-CHANTIERS DÉTAILLÉS
====================================================================

--------------------------------------------------------------------
D1 — HUB ORACLE PREMIUM À 3 PORTES
--------------------------------------------------------------------

Objectif
Transformer OracleActivity en véritable salle sacrée d’orientation, lisible et premium.

Portes à rendre réelles
1. Scanner un document
2. Importer un document
3. Créer depuis une demande / idée / sujet libre

Couche complémentaire
- Conseil des Oracles : aide l’utilisateur à formuler ce qu’il cherche avant génération.

À faire
- revoir la structure UI d’entrée ;
- créer trois zones d’action clairement distinctes ;
- donner à chaque voie une identité visuelle, un micro-texte et un son cohérents ;
- permettre le clic immédiat même si l’animation ou la vidéo d’ambiance joue encore ;
- prévoir une intervention Prométhée si l’utilisateur hésite ou reste inactif ;
- mémoriser la dernière porte utilisée pour proposer un accès rapide futur si pertinent ;
- brancher loading divin et dialogues RPG via les blocs B et C.

Résultat attendu
Le joueur comprend instantanément les trois voies et ressent que l’Oracle est un hub noble, pas un écran technique.

--------------------------------------------------------------------
D2 — SCANNER CAMERAX NON CASSÉ ET EXPLOITABLE IMMÉDIATEMENT
--------------------------------------------------------------------

Objectif
Conserver la puissance du scanner sans jamais sacrifier la preview ni l’expérience premium.

À faire
- vérifier qu’aucun overlay opaque ne casse le viewFinder ;
- garder les aides visuelles compatibles avec la lecture réelle de la preview ;
- laisser l’utilisateur scanner même si les animations d’intro sont encore en cours ;
- donner un feedback premium si le cadrage est mauvais ;
- proposer de reprendre la capture si qualité insuffisante ;
- préparer la remontée d’un score ou statut de confiance OCR ;
- ne jamais afficher un message technique brut si l’OCR lit mal le document.

Cas à couvrir
- document flou ;
- document mal cadré ;
- texte incomplet ;
- rotation / pause pendant la capture ;
- retour arrière pendant extraction.

Résultat attendu
Le scanner reste utilisable, noble et robuste sans sacrifier le cœur CameraX déjà sensible du projet.

--------------------------------------------------------------------
D3 — IMPORT DOCUMENT / IMAGE / PDF
--------------------------------------------------------------------

Objectif
Offrir une vraie voie d’import stable pour documents non scannés en direct.

À faire
- rendre l’entrée import claire et fiable ;
- prendre en charge images et PDF selon ce qui est déjà prévu dans l’existant ;
- unifier le pipeline d’analyse en aval pour éviter d’avoir un résumé de qualité inférieure juste parce que la source venait d’un import ;
- préparer le parsing propre ;
- déclencher loading divin pendant extraction et nettoyage ;
- prévoir feedback immersif si l’import contient peu de texte exploitable.

Règle UX
Importer un document ne doit pas donner l’impression d’un chemin secondaire dégradé.

Résultat attendu
Le flow import rejoint proprement le même pipeline que le scan.

--------------------------------------------------------------------
D4 — DEMANDE LIBRE / SUJET LIBRE / CRÉATION DE SAVOIR IA
--------------------------------------------------------------------

Objectif
Permettre à l’utilisateur de demander un savoir même sans document source.

À faire
- créer ou fiabiliser OraclePromptActivity si séparée ;
- permettre saisie d’un sujet, d’une idée, d’une notion, d’une question, d’un thème ;
- brancher un mode “Conseil des Oracles” où l’IA aide à préciser la demande avant génération ;
- utiliser Gemini pour produire un résumé structuré adapté à l’âge, au niveau et au contexte ;
- générer ensuite un quiz cohérent ;
- toujours faire passer par la validation du résumé comme pour les autres voies.

Cas d’usage ciblés
- “explique-moi la photosynthèse” ;
- “je veux un résumé sur la Révolution française” ;
- “aide-moi à comprendre les fractions” ;
- “je veux créer un cours sur les volcans” ;
- “je ne sais pas comment formuler mon besoin” → Conseil des Oracles.

Résultat attendu
L’Oracle devient un générateur de savoir officiel du produit, pas seulement un scanner amélioré.

--------------------------------------------------------------------
D5 — PRÉ-ANALYSE MATIÈRE, QUALITÉ ET CONTEXTE
--------------------------------------------------------------------

Objectif
Déterminer avant la sauvegarde quelle matière, quel dieu et quel temple semblent les plus cohérents.

À faire
- détecter la matière pressentie ;
- calculer une confiance matière si possible ;
- détecter si la matière semble ambiguë ;
- expliquer en langage divin pourquoi tel temple est pressenti ;
- conserver un override manuel obligatoire ;
- préparer les couleurs, le ton, le dieu et potentiellement la BGM du flow de validation.

Méta-informations utiles à préparer
- méthode d’entrée : scan / import / texte libre / conseil ;
- confiance matière ;
- qualité extraction ;
- date ;
- humeur ;
- âge ;
- profil d’apprentissage si déjà disponible ;
- éventuellement source brute ou origine.

Résultat attendu
Le joueur comprend pourquoi l’Oracle propose un temple, mais garde le contrôle final.

--------------------------------------------------------------------
D6 — GÉNÉRATION DU RÉSUMÉ STRUCTURÉ
--------------------------------------------------------------------

Objectif
Faire de chaque résumé produit par l’Oracle une fiche divine lisible, utile, agréable et cohérente.

À imposer dans la logique IA
- titre propre ;
- structure claire ;
- sections ;
- points clés si utile ;
- vocabulaire adapté à l’âge ;
- longueur maîtrisée ;
- phrases lisibles ;
- pas de bloc massif illisible ;
- pas de recyclage brut de la première ligne du document comme faux titre ;
- pas de résumé vide ou pseudo-résumé OCR brut.

À faire côté produit
- demander plusieurs propositions de titres si utile ;
- préparer des variantes de ton ou de niveau ;
- conserver la source structurée nécessaire pour l’étape de validation ;
- pouvoir régénérer ou reformuler sans casser le flow.

Résultat attendu
Le résumé sort déjà propre avant même la validation, puis devient canonique après validation.

--------------------------------------------------------------------
D7 — ÉCRAN DE VALIDATION DU RÉSUMÉ
--------------------------------------------------------------------

Objectif
Instaurer une vraie étape de contrôle humain premium avant toute sauvegarde.

Actions minimum à offrir
- Confirmer
- Éditer
- Changer temple
- Reformuler

Variantes premium à prévoir
- rendre plus simple ;
- rendre plus scolaire ;
- rendre plus court ;
- version mnémotechnique ;
- TTS sur le résumé ;
- affichage de la matière détectée et du dieu associé ;
- explication divine du temple pressenti ;
- possibilité de sélectionner un autre temple manuellement.

Protections obligatoires
- empêcher validation d’un résumé trop court ;
- empêcher validation d’un bloc brut OCR ;
- empêcher validation d’un contenu incohérent ;
- gérer le cas où la matière détectée est floue ;
- éviter les résumés trop longs qui cassent l’UI ;
- s’assurer que le titre final est propre.

Résultat attendu
L’utilisateur a confiance dans le savoir enregistré, car il l’a vu, compris, corrigé et approuvé.

--------------------------------------------------------------------
D8 — ÉDITION, REFORMULATION ET OVERRIDE MANUEL
--------------------------------------------------------------------

Objectif
Offrir le contrôle nécessaire sans transformer le flow en éditeur complexe.

À faire
- permettre une édition manuelle raisonnable du résumé ;
- permettre une reformulation rapide ;
- permettre un changement de temple/matière ;
- permettre de garder le contrôle sur le titre ;
- conserver l’historique minimal ou la distinction entre version générée et version validée si possible ;
- rendre l’ensemble cohérent avec les dialogues RPG et le TTS.

Règle produit
Le système doit être assisté par l’IA, mais la validation finale appartient au joueur.

Résultat attendu
On garde la magie IA sans sacrifier le contrôle utilisateur.

--------------------------------------------------------------------
D9 — SAUVEGARDE ATOMIQUE DU SAVOIR
--------------------------------------------------------------------

Objectif
Empêcher tout état intermédiaire sale dans la base locale.

À faire
- ne sauvegarder définitivement le CourseEntry qu’après validation ;
- stocker le résumé validé, le titre validé, la matière/temple validé ;
- stocker les métadonnées importantes ;
- empêcher les doublons involontaires liés à plusieurs tap rapides ;
- rendre la sauvegarde transactionnelle si plusieurs écritures sont nécessaires ;
- préparer le lien avec GeneratedArtifact si disponible plus tard ;
- prévoir un état retour arrière propre si l’utilisateur annule avant validation.

Métadonnées recommandées
- sourceType ;
- inputMethod ;
- detectedSubject ;
- validatedSubject ;
- confidence ;
- createdAt ;
- moodAtCreation ;
- ageAtCreation ;
- learningProfileAtCreation ;
- rawSourceReference éventuelle.

Résultat attendu
La base ne contient que des savoirs validés et cohérents.

--------------------------------------------------------------------
D10 — GÉNÉRATION DU QUIZ À PARTIR DU RÉSUMÉ VALIDÉ
--------------------------------------------------------------------

Objectif
Verrouiller le principe selon lequel le quiz doit être généré à partir du savoir canonique validé, pas depuis un état de travail intermédiaire.

À faire
- faire dépendre explicitement la génération du quiz du résumé validé ;
- empêcher la génération directe depuis la matière première sale ;
- préparer le passage propre vers le bloc E ;
- transmettre les bonnes métadonnées au générateur de quiz ;
- conserver la cohérence entre résumé vu par l’utilisateur et quiz qu’il reçoit.

Règle absolue
Si le résumé a été modifié ou reformulé, c’est cette version-là qui pilote le quiz.

Résultat attendu
Le joueur ne ressent plus de décalage entre ce qu’il lit et ce qu’on lui demande ensuite.

--------------------------------------------------------------------
D11 — ERREURS, LIMITES ET RETOURS DIÉGÉTIQUES
--------------------------------------------------------------------

Objectif
Rendre tous les cas limites immersifs et compréhensibles.

À couvrir
- document illisible ;
- peu de texte détecté ;
- matière ambiguë ;
- génération de résumé incomplète ;
- échec temporaire réseau ;
- quota atteint ;
- import vide ;
- annulation utilisateur ;
- retour arrière pendant chargement.

Règle
Aucun message technique brut.
Toujours une reformulation diégétique cohérente avec le dieu ou l’Oracle.

Exemples d’intention
- “Les Oracles perçoivent ce parchemin avec hésitation.”
- “Le rituel n’a pas encore révélé un savoir assez clair.”
- “Le dieu concerné se repose avant de reprendre cette invocation.”

Résultat attendu
Même l’échec reste premium et cohérent avec RéviZeus.
