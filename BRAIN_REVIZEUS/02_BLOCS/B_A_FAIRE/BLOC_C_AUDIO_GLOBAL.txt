====================================================================
0 — RÔLE DE CE BLOC
====================================================================

Ce bloc sert à industrialiser toute la couche audiovisuelle transverse de RéviZeus.
Il ne s’agit pas seulement de “faire du son plus propre” ou de lire du texte à voix haute.
Il s’agit de transformer tous les systèmes sensibles du jeu pour qu’ils reposent sur une base cohérente, stable, immersive et réutilisable : musiques, transitions, TTS, ducking, loadings divins, vidéos de fond, gel image fixe, lifecycle, reprise après interruption, et compatibilité avec les blocs cœur du jeu.

Sans ce bloc :
- les activités peuvent se couper ou se chevaucher musicalement ;
- le TTS risque de gêner la BGM ou d’être imprévisible ;
- les futures actions IA longues donnent une sensation de bug ou de vide ;
- les fonds vidéo restent dispersés et non homogènes ;
- l’Oracle, les quiz, les résultats, les actions divines et les futurs écrans premium héritent d’une base instable ;
- les futurs prompts donnés à une IA risquent de recoder localement le son, la voix, les chargements et les vidéos au lieu de s’appuyer sur une couche centrale.

Ce bloc est donc le chantier qui transforme l’audio et l’habillage animé en infrastructure premium partagée.

====================================================================
1 — OBJECTIF FINAL
====================================================================

Créer une couche audiovisuelle transverse capable de gérer, de façon homogène et réutilisable :
- la mémoire musicale entre écrans ;
- les transitions musicales différées et sûres ;
- le ducking de la BGM sous la voix ;
- la lecture TTS sur tous les contenus longs utiles ;
- l’arrêt, la reprise et l’abandon propres du TTS ;
- un bouton speaker universel cohérent ;
- un LoadingDivineDialog central, noble, immersif, réutilisable ;
- des textes de chargement contextuels par dieu et par action ;
- la traduction immersive des délais, indisponibilités et quotas en fatigue divine ;
- un helper universel vidéo -> image fixe ;
- le pattern raw xxx_animated -> drawable xxx ;
- un fallback instantané si la vidéo manque ;
- l’absence de son parasite sur les vidéos décoratives ;
- un comportement propre dans le lifecycle ;
- une base prête pour Oracle, quiz, résultats, actions divines, écran info, intro, temples, Jukebox et futurs contenus.

À la fin du bloc C, RéviZeus doit disposer d’un socle audio/voix/loading/background suffisamment robuste pour que les blocs D à O n’aient plus à bricoler localement leur audiovisuel.

--------------------------------------------------------------------
C1 — SOUNDMANAGER GLOBAL ET MÉMOIRE MUSICALE
--------------------------------------------------------------------

Objectif
Faire de SoundManager la façade audio réellement fiable du projet, au lieu d’un simple lanceur de musiques.

À créer ou fiabiliser
- rememberMusic(resId, loop = true)
- getRememberedMusicResId()
- playMusicDelayed(resId, loop = true, delayMs = 300)
- resumeRememberedMusicDelayed(delayMs = 300)
- cancelDelayedMusic()
- stopMusic() / stopAllIfNeeded() si déjà présents
- garde-fous contre doubles play successifs
- gestion propre du changement d’activité rapide
- gestion sûre après pause / reprise / back / finish

Responsabilités attendues
- mémoriser la musique de référence d’un écran ou d’un état ;
- éviter les superpositions ou écrasements sauvages ;
- lisser les transitions entre écrans ;
- empêcher un ancien écran de reprendre sa musique après destruction ;
- donner un comportement cohérent entre Dashboard, Temples, Oracle, Quiz, Résultats, Jukebox et écrans futurs.

Améliorations à intégrer directement
- journal audio léger pour debug ;
- stratégie simple pour savoir si une musique est déjà en cours ;
- garde-fou contre restart identique inutile ;
- priorité simple entre BGM standard, BGM de résultat, Jukebox, loading sonore et TTS ;
- base extensible pour variations par humeur / temple / rang plus tard.

Livrables
- SoundManager stabilisé ;
- transitions audio plus propres ;
- mémoire musicale réellement exploitable par tout le jeu.

--------------------------------------------------------------------
C2 — TTS GLOBAL UNIFIÉ AVEC SPEAKERTTSHELPER
--------------------------------------------------------------------

Objectif
Créer un moteur TTS central réutilisable partout, au lieu d’avoir des implémentations locales disparates.

À créer ou fiabiliser
- SpeakerTtsHelper.kt global ;
- API simple pour speak(text), stop(), pause si possible, resume si possible ;
- gestion du lifecycle ;
- nettoyage des listeners ;
- callback de fin de lecture ;
- état de lecture isSpeaking ;
- possibilité de lier un bouton speaker à différents écrans sans duplication ;
- arrêt propre si l’activité change ou se ferme.

Écrans et contenus cibles minimum
- résumés ;
- validation de résumé ;
- questions de quiz ;
- corrections ;
- résultats ;
- dialogues longs critiques quand pertinent ;
- futur écran info ;
- futurs artefacts textuels (poème, traduction, reformulations).

Améliorations à intégrer directement
- support vitesse de lecture configurable ;
- paramètre activation/désactivation TTS ;
- structure permettant plus tard des voix différenciées ou styles de lecture ;
- structure permettant plus tard mise en évidence du texte lu ;
- stop immédiat à tout moment ;
- comportement idempotent si l’utilisateur spamme le speaker.

Livrables
- moteur TTS central fiable ;
- API simple pour les futurs blocs ;
- suppression du bricolage TTS local.

--------------------------------------------------------------------
C3 — BOUTON SPEAKER UNIVERSEL ET USAGES ÉCRAN
--------------------------------------------------------------------

Objectif
Éviter que chaque écran invente sa propre logique de bouton speaker, son placement, son état et son comportement.

À faire
- définir le comportement standard du speaker ;
- homogénéiser l’icône, la taille et le feedback d’état ;
- prévoir état lecture / arrêt ;
- éviter les icônes coupées, trop zoomées ou mal cadrées ;
- prévoir intégration simple dans les layouts XML existants ;
- garder une logique de clic cohérente entre résumé, quiz, résultat et dialogues.

Règles UX
- un appui démarre la lecture ;
- un second appui stoppe ;
- la lecture doit se couper proprement quand l’utilisateur change d’écran ;
- la BGM doit être duckée pendant la lecture ;
- le bouton ne doit pas être caché, tronqué ou disproportionné ;
- le composant doit rester premium et lisible en portrait.

Améliorations à intégrer directement
- état visuel lecture en cours ;
- animation légère si pertinent ;
- préparation pour highlight du segment lu ;
- cohérence avec la charte audio globale.

Livrables
- comportement speaker unifié ;
- intégration cohérente des écrans prioritaires.

--------------------------------------------------------------------
C4 — DUCKING AUDIO ET ARBITRAGE ENTRE BGM / TTS / SFX
--------------------------------------------------------------------

Objectif
Faire du ducking un comportement central, invisible, fiable et agréable.

À mettre en place
- baisse automatique de la BGM pendant la voix ;
- restauration à la fin de la lecture ;
- restauration sûre en cas d’arrêt manuel ;
- protection si l’écran change pendant la lecture ;
- éviter toute remontée brutale ou décalée ;
- éviter les cas où la voix et la musique se battent ;
- éviter qu’un SFX critique soit masqué ou que la voix masque tout.

Cas à tester impérativement
- lecture d’un résumé puis arrêt manuel ;
- lecture d’une question puis changement de question ;
- lecture d’une correction puis retour en arrière ;
- lecture avec changement d’activité ;
- lecture avec BGM spécifique d’écran ;
- lecture pendant ou après un loading ;
- lecture dans un écran déjà riche en SFX.

Améliorations à intégrer directement
- intensité de ducking configurable ;
- méthode centrale au lieu de bricolage par écran ;
- stratégie de priorité minimale : TTS > BGM, mais sans tuer tous les SFX utiles ;
- base future pour accessibilité et profils d’apprentissage.

Livrables
- ducking réellement stable ;
- coexistence propre entre musique et voix.

C5 — LOADING DIVIN UNIVERSEL
--------------------------------------------------------------------

Objectif
Faire des attentes IA un moment immersif, noble, lisible et sécurisé.

À créer ou fiabiliser
- LoadingDivineDialog.kt central ;
- une seule instance à la fois ;
- helpers showLoading()/hideLoading() dans BaseActivity ou BaseGameActivity ;
- gestion non cancelable si critique ;
- gestion cancelable si non critique selon contexte ;
- fermeture sûre même si retour erreur ;
- aucun écran noir ou blanc entre loading et résultat ;
- aucune fuite de fenêtre.

Cas d’usage prioritaires
- génération de résumé ;
- génération de quiz ;
- OCR lourd ;
- traduction Hermès ;
- dessin Aphrodite ;
- Lyre d’Apollon ;
- corrections IA longues ;
- futures générations de musique ou artefacts créatifs.

Améliorations à intégrer directement
- messages aléatoires ou contextuels ;
- style de message par dieu ;
- micro animation portrait / chibi / particules ;
- SFX de boucle légère, stoppée proprement ;
- reprise automatique de la BGM précédente après fermeture ;
- texte d’attente non technique ;
- timeout doux avec message immersif si l’attente dure.

Livrables
- loading central robuste ;
- disparition des attentes “muettes” ;
- base premium pour tous les blocs IA.

--------------------------------------------------------------------
C6 — FATIGUE DIVINE ET INDISPONIBILITÉS AUDIO/IA
--------------------------------------------------------------------

Objectif
Transformer les indisponibilités, délais et quotas en retours diégétiques cohérents au lieu d’erreurs sèches.

À couvrir
- quota image atteint ;
- quota musique atteint ;
- quota traduction atteint ;
- génération temporairement indisponible ;
- service lent ou interrompu ;
- réessai conseillé plus tard ;
- action longue encore en cours.

Règles
- ne pas mentir techniquement ;
- ne pas afficher le jargon brut ;
- toujours proposer une formulation cohérente avec le dieu ;
- si utile, donner un délai estimé ou une alternative ;
- ne pas laisser le joueur sans explication ni issue.

Exemples attendus
- Apollon accorde encore sa lyre, reviens bientôt ;
- Aphrodite est fatiguée, sa vision du savoir se reposera avant une nouvelle peinture ;
- Les vents d’Hermès sont troublés, la traduction devra patienter ;
- Les Oracles poursuivent encore leur lecture, attends un instant.

Livrables
- helpers ou référentiel de textes immersifs ;
- usage intégré au loading ou aux dialogs RPG si disponibles.

C7 — VIDEOBACKGROUNDHELPER ET PATTERN VIDÉO -> IMAGE FIXE
--------------------------------------------------------------------

Objectif
Uniformiser définitivement le comportement des écrans à fond animé.

À créer ou fiabiliser

Convention de ressource à respecter
- raw : xxx_animated.mp4
- drawable still : xxx
- exemple : bg_oracle_animated.mp4 -> bg_oracle.png ;
- exemple : bg_resultat_animated.mp4 -> bg_resultat.png ;
- exemple : bg_avatar_x_animated.mp4 -> still correspondante si prévue.

Écrans prioritaires

- OracleActivity ;

- écrans info ;

Améliorations à intégrer directement

- journal debug si ressource manquante ;
- pas de son parasite sur vidéo décorative ;
- préparation pour plusieurs résolutions si besoin futur ;

Livrables
- helper vidéo central ;
