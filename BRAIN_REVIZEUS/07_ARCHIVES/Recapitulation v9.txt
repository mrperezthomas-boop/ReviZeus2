# 1. Règles de reprise obligatoires

## Règles d’architecture

* conserver **100 % de l’architecture générale**
* ne faire que des **insertions / ajustements localisés**
* ne jamais casser les flux existants
* toujours renvoyer les **fichiers complets** si modification

## Règles UX

* orientation **portrait obligatoire**
* tout dialogue important doit être en **style RPG immersif**, jamais en popup Android basique
* la BGM continue pendant les dialogues
* `sfx_dialogue_blip` doit s’arrêter dès que :

  * le texte disparaît
  * le texte est skip
  * l’écran change
  * `onPause()`
  * `onDestroy()`

## Règles ressources

* les ressources principales existent déjà dans `res/drawable` et `res/raw`
* avant de créer une nouvelle ressource, vérifier si l’existant couvre déjà le besoin
* si une nouvelle ressource est vraiment nécessaire :

  * nom exact
  * rôle
  * écrans concernés
  * connexions code
  * prompt conforme à la DA

## 3.1 `BaseActivity.kt`

### Rôle

Classe de base commune pour les activités principales.

### Dépendances

* Android Activity lifecycle
* souvent `SoundManager`
* helpers visuels / vibration / UI globale

### Risques

* si on met trop de logique métier ici, on pollue toute l’app
* toute régression dans `BaseActivity` touche beaucoup d’écrans

### Ressources utilisées

* aucune ressource métier spécifique à elle seule
* peut piloter des SFX communs

### Points exacts où brancher des futures features

# 6. Conclusion de reprise

RéviZeus est déjà un projet **très avancé**.
La bonne manière de le continuer n’est plus de “coder au hasard”, mais de le traiter comme un vrai projet de production :

```text
fichier par fichier
dépendance par dépendance
en protégeant les flux existants
```

La règle d’or :

```text
stabiliser d’abord
puis enrichir
sans jamais casser l’immersion divine
```

Si tu veux, au prochain message je peux te faire le **niveau ultime** :
un **tableau d’intervention par feature**, qui dit pour chaque future amélioration :

* fichiers exacts à modifier
* ordre d’édition
* risques de casse
* tests à faire après modification.
